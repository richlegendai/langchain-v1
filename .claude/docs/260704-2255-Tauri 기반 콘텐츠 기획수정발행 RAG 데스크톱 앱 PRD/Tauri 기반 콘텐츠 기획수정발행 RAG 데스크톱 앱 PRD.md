# Tauri 기반 콘텐츠 기획·수정·발행 RAG 데스크톱 앱 PRD

## 1. 문서 목적

이 문서는 Tauri 기반 콘텐츠 기획·수정·발행 RAG 데스크톱 앱의 제품 요구사항을 정의합니다. 기준 자료는 `.claude/docs/260704-2244-Tauri 기반 콘텐츠 기획수정발행 RAG 데스크톱 앱 워크플로우/`의 워크플로우 문서와 현재 `README.md`입니다.

## 2. 제품 한 줄 정의

프로젝트 내부에 저장한 로컬 콘텐츠 자료와 발행 기준을 검색 증강 생성에 활용하고, 로컬 CLI 기반 LLM 도구를 호출해 콘텐츠 후보 생성, 사용자 선택, 수정, 검수, 발행 준비까지 처리하는 Tauri 기반 콘텐츠 운영 데스크톱 앱입니다.

## 3. 배경과 문제

콘텐츠 운영자는 기존 콘텐츠, CTA 기준, 톤앤매너 예시, 발행 체크리스트를 참고해 새 콘텐츠를 반복 작성합니다. 이 과정은 자료 검색, 초안 작성, 후보 비교, 수정 요청, 검수, 발행용 파일 저장으로 나뉘며 수동 작업이 많습니다.

이 프로젝트는 외부 API 기반 LLM 호출보다 로컬에 로그인된 Claude Code CLI, Codex CLI, Gemini CLI 같은 도구를 직접 활용하는 방향을 전제로 합니다. 따라서 브라우저 기반 웹앱보다 로컬 명령 실행, 파일 접근, 로컬 데이터베이스 관리가 가능한 Tauri 데스크톱 앱이 제품 형태에 적합합니다.

## 4. 목표

- 프로젝트 내부 `data/sources/` 자료를 기반으로 RAG 검색을 수행합니다.
- 사용자가 입력한 주제와 키워드에 맞는 콘텐츠 후보 3개를 생성합니다.
- 사용자가 후보, 말투, 수정 요청을 선택해 반복 수정할 수 있게 합니다.
- 검수 체크리스트를 통해 키워드, CTA, 금지 표현, 발행 기준을 확인합니다.
- 발행용 Markdown 또는 HTML 파일과 작업 이력을 로컬에 저장합니다.
- CLI 실행 결과와 실패 원인을 사용자에게 명확히 보여줍니다.

## 5. 비목표

- 프로젝트 밖의 Obsidian vault, 다운로드 폴더, 개인 문서 폴더를 자동 수집하지 않습니다.
- 완전 자동 발행을 MVP 범위에 포함하지 않습니다.
- 여러 LLM CLI의 무중단 운영이나 상용 배포 안정성 보장을 MVP 범위에 포함하지 않습니다.
- 클라우드 기반 사용자 계정, 팀 협업, 결제, 권한 관리는 초기 범위에 포함하지 않습니다.

## 6. 대상 사용자

| 사용자 | 니즈 |
| --- | --- |
| 콘텐츠 운영자 | 기존 콘텐츠와 발행 기준을 반영한 초안을 빠르게 만들고 싶습니다. |
| 블로그·SNS 작성자 | 여러 후보를 비교하고 말투를 바꿔가며 발행용 문서를 다듬고 싶습니다. |
| 로컬 LLM CLI 사용자 | 이미 로그인된 CLI 도구를 앱 안의 워크플로우에서 재사용하고 싶습니다. |

## 7. 핵심 사용 시나리오

### 7.1 콘텐츠 후보 생성

1. 사용자가 주제와 키워드를 입력합니다.
2. 앱이 `data/sources/`의 프로젝트 내부 자료를 검색합니다.
3. 검색 결과와 발행 기준을 프롬프트에 결합합니다.
4. CLI LLM Adapter가 선택된 로컬 CLI 도구를 호출합니다.
5. 앱이 콘텐츠 후보 3개를 카드 형태로 표시합니다.

### 7.2 후보 선택과 수정

1. 사용자가 후보 1개를 선택합니다.
2. 사용자가 말투 옵션과 수정 요청을 입력합니다.
3. LangGraph State에 선택값과 수정 요청을 저장합니다.
4. CLI LLM Adapter가 선택 후보를 기준으로 수정본을 생성합니다.
5. 앱이 수정본과 이전 버전을 비교 가능한 상태로 표시합니다.

### 7.3 검수와 발행 준비

1. 앱이 키워드, CTA, 금지 표현, 발행 기준 체크리스트를 실행합니다.
2. 검수 결과를 통과, 주의, 실패 상태로 표시합니다.
3. 사용자가 최종 승인하면 발행용 Markdown 또는 HTML을 저장합니다.
4. SQLite에 초안, 선택 이력, 수정 요청, 발행 상태, CLI 실행 로그를 저장합니다.

## 8. 데이터 범위

RAG 검색과 콘텐츠 생성의 원천 자료는 프로젝트 내부 파일만 대상으로 합니다.

| 경로 | 용도 |
| --- | --- |
| `data/sources/content/` | 기존 콘텐츠, 블로그 초안, SNS 초안, 뉴스레터 초안 |
| `data/sources/guidelines/` | CTA 기준, 톤앤매너 예시, 발행 기준, 금지 표현, 검수 체크리스트 |
| `data/sources/references/` | 참고 자료, 제품 설명, 고객 사례, 조사 메모 |
| `data/chroma_db/` | Chroma 벡터 데이터 저장소 |
| `data/app.sqlite` | 초안, 수정 이력, 상태, CLI 실행 로그 저장소 |

지원 파일 형식의 초기 범위는 Markdown, HTML, TXT, CSV입니다.

## 9. 제품 요구사항

### 9.1 콘텐츠 자료 인덱싱

- 앱은 `data/sources/` 하위 파일을 스캔할 수 있어야 합니다.
- 앱은 지원 파일 형식을 Chunking과 Embedding 처리할 수 있어야 합니다.
- 앱은 처리 결과를 Chroma에 저장하고 검색할 수 있어야 합니다.
- 앱은 인덱싱 성공, 실패, 제외 파일 수를 사용자에게 보여줘야 합니다.

### 9.2 주제 입력

- 사용자는 콘텐츠 주제와 키워드를 입력할 수 있어야 합니다.
- 사용자는 발행 채널을 선택할 수 있어야 합니다.
- 사용자는 후보 생성에 사용할 LLM CLI provider를 선택할 수 있어야 합니다.
- 입력값이 부족하면 앱은 생성 실행 전에 부족한 항목을 표시해야 합니다.

### 9.3 후보 생성

- 앱은 검색된 프로젝트 내부 자료와 발행 기준을 프롬프트에 포함해야 합니다.
- 앱은 콘텐츠 후보 3개를 생성해야 합니다.
- 각 후보는 제목, 요약, 본문, 추천 채널, 참고한 자료 요약을 포함해야 합니다.
- CLI 호출 실패 시 stderr, exit code, timeout 여부를 저장하고 화면에 표시해야 합니다.

### 9.4 후보 선택과 수정

- 사용자는 후보 1개를 선택할 수 있어야 합니다.
- 사용자는 말투 옵션과 자유 수정 요청을 입력할 수 있어야 합니다.
- 앱은 선택 후보와 수정 요청을 기준으로 수정본을 생성해야 합니다.
- 앱은 수정 전후 내용을 비교할 수 있게 표시해야 합니다.
- 사용자는 수정 요청을 반복할 수 있어야 합니다.

### 9.5 검수

- 앱은 키워드 포함 여부를 확인해야 합니다.
- 앱은 CTA 기준 반영 여부를 확인해야 합니다.
- 앱은 금지 표현 포함 여부를 확인해야 합니다.
- 앱은 발행 기준 체크리스트 결과를 항목별로 표시해야 합니다.
- 검수 실패 항목은 최종 발행 준비 상태로 전환하기 전에 사용자에게 노출되어야 합니다.

### 9.6 발행 준비

- 사용자는 최종본을 Markdown 또는 HTML로 저장할 수 있어야 합니다.
- 앱은 최종본 저장 경로를 표시해야 합니다.
- 앱은 콘텐츠 상태를 `draft`, `reviewing`, `ready_to_publish` 중 하나로 관리해야 합니다.
- 앱은 초안, 수정 이력, 선택 이력, 발행 상태를 SQLite에 저장해야 합니다.

### 9.7 CLI LLM Adapter

- Adapter는 `claude`, `codex`, `gemini` provider를 동일한 인터페이스로 호출할 수 있어야 합니다.
- Adapter는 timeout을 적용해야 합니다.
- Adapter는 stdout, stderr, exit code를 캡처해야 합니다.
- Adapter는 민감정보 마스킹을 수행한 뒤 로그를 저장해야 합니다.
- Adapter는 지원하지 않는 provider 입력을 명시적 오류로 처리해야 합니다.

## 11. 워크플로우

```text
사용자 주제 입력
-> Tauri React UI
-> FastAPI Agent Server 호출
-> LangGraph workflow 시작
-> Chroma에서 프로젝트 내부 콘텐츠와 발행 기준 검색
-> CLI LLM Adapter가 로컬 LLM CLI 호출
-> 콘텐츠 후보 3개 생성
-> Tauri 화면에 후보 표시
-> 사용자 후보 선택, 말투 선택, 수정 요청 입력
-> LangGraph State에 선택값 저장
-> CLI LLM으로 수정본 재생성
-> 검수 체크리스트 실행
-> 발행용 Markdown 또는 HTML 생성
-> SQLite에 초안, 수정 이력, 발행 상태 저장
```

## 12. 주요 API 요구사항

| Endpoint | 목적 | 주요 입력 | 주요 출력 |
| --- | --- | --- | --- |
| `POST /generate` | 콘텐츠 후보 생성 | 주제, 키워드, 채널, provider | 후보 3개, 검색 요약, 실행 로그 ID |
| `POST /revise` | 선택 후보 수정 | draft ID, 후보 ID, 말투, 수정 요청 | 수정본, 변경 요약, 실행 로그 ID |
| `POST /approve` | 최종 승인 | draft ID, 출력 형식 | 저장 경로, 상태 |
| `GET /drafts` | 초안 목록 조회 | 상태, 기간 | 초안 목록 |
| `GET /drafts/{id}` | 초안 상세 조회 | draft ID | 본문, 이력, 검수 결과 |
| `POST /index` | 자료 인덱싱 | 대상 경로, 재색인 여부 | 처리 파일 수, 실패 목록 |

## 14. 비기능 요구사항

- 로컬 파일 접근은 프로젝트 내부 `data/` 범위를 기본값으로 제한합니다.
- CLI 호출 실패는 조용히 무시하지 않고 화면과 로그에 표시합니다.
- 사용자가 긴 작업 상태를 이해할 수 있도록 생성, 수정, 검수, 저장 단계를 구분해 표시합니다.
- SQLite와 Chroma 파일은 기본적으로 프로젝트 내부에 저장합니다.
- 민감정보는 CLI 실행 로그 저장 전에 마스킹합니다.
- MVP 단계에서는 네트워크 연결 없이도 로컬 자료 검색과 로컬 CLI 호출 중심으로 동작해야 합니다.

## 15. 수용 기준

- 사용자는 `data/sources/`에 넣은 자료를 인덱싱할 수 있습니다.
- 사용자는 주제와 키워드를 입력해 콘텐츠 후보 3개를 생성할 수 있습니다.
- 후보 생성 결과는 프로젝트 내부 자료를 참고한 흔적을 포함합니다.
- 사용자는 후보를 선택하고 수정 요청을 입력해 수정본을 생성할 수 있습니다.
- 검수 체크리스트 결과가 화면에 표시됩니다.
- 최종본은 Markdown 또는 HTML 파일로 저장됩니다.
- SQLite에는 초안, 수정 이력, 상태, CLI 실행 로그가 저장됩니다.
- CLI 실패 시 실패 원인이 사용자에게 보이고 로그에 남습니다.

## 16. MVP 범위

1. 프로젝트 내부 자료 폴더 구조 생성과 인덱싱
2. FastAPI Agent Server 기본 실행
3. Chroma 기반 검색
4. CLI LLM Adapter의 `codex` 또는 `claude` 호출
5. React UI의 주제 입력, 후보 카드, 수정 요청, 검수 결과, 발행 상태
6. SQLite 기반 초안과 실행 로그 저장
7. Markdown 또는 HTML 최종본 저장

| 검수 기준 불명확 | 발행 품질 편차 | `data/sources/guidelines/`의 체크리스트를 명시적으로 관리 |

## 19. 미해결 질문

- MVP의 첫 provider를 `codex`로 둘지 `claude`로 둘지 결정이 필요합니다.
- 발행용 HTML 템플릿을 고정할지 사용자 설정으로 둘지 결정이 필요합니다.
